# 에이전트 시스템 설계

## 에이전트 시스템의 핵심 구성요소

효과적인 에이전트 기반 시스템을 설계하려면 에이전트가 임무를 잘 수행하도록 돕는 핵심 구성요소를 깊게 이해해야 한다. 각 구성요소는 에이전트의 능력, 효율, 적응성을 결정한다.

사용자 → 오케스트레이션 → 모델, 도구, 메모리, 지식베이스

## 모델 선택

모든 에이전트 기반 시스템의 중심에는 모델이 있다. 모델은 에이전트의 의사결정, 상호작용, 학습 역량을 결정한다. 그렇기에 에이전트 시스템 구축은 모델을 선택하는 데서 시작한다.

모델 선택은 작업 복잡도 평가에서 시작한다. GPT나 클로드 같은 대형 파운데이션 모델은 개방형 환경에서 컨텍스트 이해, 유연한 추론, 창의적 생성이 필요한 에이전트에 적합하다. 이런 모델은 일반화가 뛰어나고 모호성, 문맥적 뉘앙스, 다단계 작업에 강하다.

소형 모델은 정의가 명확한 반복 작업에 적합한 경우가 많다. 로컬 하드웨어에서 효율적으로 실행되고 응답이 빠르며 배포 및 운영 비용이 낮다. 고객 지원, 정보 검색, 데이터 라벨링처럼 구조화된 환경에서 유용하다. 실시간 응답성이나 자원 제약이 중요한 경우 작은 모델이 실용성 덕분에 큰 모델을 능가하기도 한다.

일반적인 언어 과제, 빠른 프로토타이핑, 도메인 특화 정밀도가 중요하지 않은 상황에서는 사전학습 모델이 적합하며, 의료,법률, 기술 지원과 같은 특수 도메인에서는 맞춤 학습 모델이 상당한 이점을 제공한다.

모델 선택은 한번 결정하고 끝날 문제가 아니라 에이전트 역량, 사용자 요구, 인프라가 진화함에 따라 지속적으로 재검토해야 한다. 일반성 대 전문성, 성능 대 비용, 단순성 대 확장성 같은 트레이드오프를 저울질해서 결정해야 한다.

## 도구

에이전트 기반 시스템에서 도구는 에이전트가 특정 작업을 수행하고 문제를 해결할 수 있게 한다. 도구는 에이전트의 기능적 빌딩 블록으로 작업 실행과 사용자 및 다른 시스템과 상호작용을 가능하게 한다. 도구는 크게 3가지로 나눠진다.

- 로컬 도구 : 에이전트가 외부 의존성 없이 내부 로직과 계산만으로 작업을 수행한다. 주로 규칙 기반이거나 사전에 정의된 함수를 실행하는 방식이다.
- API 기반 도구 : 에이전트가 외부 서비스나 데이터 소스와 상호작용하도록 한다. 실시간 데이터를 조회하거나 외부 시스템을 활용해 로컬 환경 너머의 기능을 수행할 수 있다.
- MCP : 표준 스키마를 통해 사용자 프로필, 대화 이력, 세계 상태, 작업 메타데이터 같은 구조화된 실시간 컨텍스트를 프롬프트에 직접 전달한다.

## 메모리

메모리는 에이전트가 정보를 저장하고 검색할 수 있게 하는 핵심 구성 요소이다. 이를 통해 에이전트는 컨텍스트를 유지하고 과거 상호작용에서 학습하며 시간이 지남에 따라 더 나은 의사결정을 내릴 수 있다.

단기 메모리는 에이전트가 현재 작업이나 대화와 관련된 정보를 저장하고 관리하는 능력을 의미한다. 이러한 유형의 메모리는 상호작용 중 컨텍스트를 유지하는 데 사용되며 에이전트가 실시간으로 일관된 의사결정을 내릴 수 있도록 돕는다.

장기 메모리는 에이전트가 오랜 기간에 걸쳐 지식과 경험을 저장하도록 하여 과거 정보를 바탕으로 향후 행동을 결정할 수 있게 한다. 이는 시간이 지남에 따라 점차 발전해야 하거나 사용자 선호도에 기반한 개인화된 경험을 제공해야 하는 에이전트에 특히 중요하다.

## 오케스트레이션

오케스트레이션은 여러 스킬을 배치하고 상황에 따라 실행하며 전체 과정을 감독하는 논리로 각 단계가 다음 단계로 자연스럽게 이어지며 명확한 목표를 이루도록 한다. 오케스트레이션의 핵심은 도구나 스킬, 호출의 가능한 순서를 평가하고 그 결과를 예측하며 여러 단계를 거치는 작업에서 가장 성공 가능성이 높은 경로를 선택하는데 있다.

## 설계 트레이드 오프

에이전트 기반 시스템을 설계할 때는 성능, 확장성, 신뢰성, 비용 사이의 다양한 트레이드 오프를 균형있게 조정해야 한다. 이러한 요소들은 실제 환경에서 에이전트가 어떻게 작동하고 성능을 발휘하는지를 결정하므로 개발자는 전략적인 선택을 내려야 한다.

### 성능: 속도와 정확도의 균형

에이전트 설계에서 중요한 트레이드오프 하나는 속도와 정확도의 균형이다. 높은 성능은 정보를 빠르게 처리하고 의사결정을 내리며 작업을 수행할 수 있지만 그만큼 정밀도가 떨어질 수 있다. 반대로 정확도를 높이려면 복잡한 모델이나 계산 집약적인 기법이 필요해 전체 속도가 느려질 수 있다.

속도 우선 → 자율주행, 트레이딩 시스템과 같이 실시간 의사결정이 중요한 환경
정확도 우선 → 법률 분석, 의료 진단과 같이 정확성이 절대적인 영역
하이브리드 → 에이전트가 먼저 빠르고 대략적인 결과를 제시한 뒤 추가로 시간과 데이터를 활용해 이를 정교하게 보완하는 방식. 추천 시스템이나 진단 시스템에서 흔히 사용된다.

### 확장성: 에이전트 시스템의 엔지니어링적 확장

현대의 에이전트 시스템은 딥러닝 모델과 실시간 처리에 크게 의존하기 때문에 확장성은 중요한 기술적 과제다. 시스템이 커질수록 데이터량, 동시 처리 작업 수, 연산 리소스의 효율적인 관리가 핵심이다.

대규모 에이전트 시스템에서는 특히 지연시간이 문제로 떠오른다. 에이전트가 실시간 또는 준실시간 환경에서 상호작용해야 하는 경우 최소 지연으로 작동할 수 있도록 GPU 작업을 효율적으로 스케줄링해야 한다. 분산 시스템 전반에 GPU 작업을 균형있게 배분하려면 지연을 줄이고 높은 부하에서도 시스템이 안정적으로 작동하도록 한다.

비동기 작업 실행이 효과적인 전략이다. 이전 작업의 완료를 기다리지 않고 GPU 작업을 병렬로 처리해 GPU 자원 활용도를 극대화하고 작업 간 유휴 시간을 최소화한다.

또 다른 전략으로는 동적 로드 밸런싱이 있다. 활용률이 낮은 GPU로 작업을 분산시켜 특정 GPU에 병목이 생기지 않도록 한다. 

수평 확장 : 시스템의 처리 능력을 높이기 위해 GPU 노드를 추가하는 방식

### 신뢰성: 견고하고 일관된 에이전트

신뢰성은 에이전트가 일정한 기간 동안 일관되고 정확하게 작업을 수행할 수 있는 능력을 의미한다. 신뢰할 수 있는 에이전트는 예상된 상황뿐 아니라 예기치 못한 조건에서도 문제 없이 작동해야 하며 이를 통해 사용자와 이해관계자의 높은 신뢰를 확보한다. 다만 신뢰성을 향상시키려면 시스템 복잡성, 비용, 개발 기간 측면에서 일정한 절충이 필요하다.

**장애 허용**

신뢰성의 한 가지 핵심 요소는 에이전트가 오류나 예기치 못한 사건을 감지하고 비정상 종료나 예측 불가능한 오작동 없이 이를 처리할 수 있도록 하는 것이다. 이를 위해 장애 허용 메커니즘을 구축해야 한다. 장애 허용 시스템은 네트워크 중단, 하드웨어 장애 등의 문제를 감지하고 정상 상태로 복구할 수 있어야 한다. 이러한 시스템은 일반적으로 중복 구조를 사용한다. 주요 구성 요소나 프로세스를 복제해 시스템 일부가 고장나더라도 전체 성능이나 서비스가 중단되지 않도록 보장한다.

### 비용: 성능과 지출의 균형

비용은 매우 중요한 고려요소다. 에이전트를 개발, 배포, 유지하는데 드는 비용은 기대되는 이익과 투자 수익률을 기준으로 평가해야 한다. 비용은 모델 복잡도, 인프라, 확장성과 관련된 의사결정 전반에 영향을 미친다.

**개발 비용**

고도화된 에이전트를 개발하는 과정은 비용이 많이 든다. 특히 대규모 데이터셋, 전문 인력, 막대한 연산 자원이 필요한 머신러닝 모델을 사용하는 경우 더욱 그렇다. 여기에 반복적인 설계, 테스트, 최적화 과정이 필요하므로 개발 비용은 더욱 증가한다.

**운영 비용**

배포 이후 에이전트를 실해아는 데 드는 운영 비용이 상당할 수 있다. 특히 실시간 의사 결정이나 지속적인 데이터 처리가 필요한 시스템의 경우 고성능 연산이 필수적이기 때문이다. 

또한 방대한 데이터를 처리하거나 대규모 메모리를 유지하는 에이전트는 데이터 저장 및 대역폭 비용을 크게 잡아 먹는다.

**비용 대비 가치**

에이전트 시스템의 비용은 그 가치로 정당화되어야 한다. 중요도가 낮은 작업이라면 단순하고 저비용의 에이전트를 사용하는 편이 더 합리적이지만 멈추거나 중단되면 안되는 작업이라면 복잡하고 고성능의 에이전트에 과감히 투자해야 한다. 비용을 최적화하기 위한 대표적인 3가지 전략을 제시한다.

- 경량 모델 : 특정 작업에서 규칙 기반 시스템이 딥러닝 모델과 유사한 성능을 낼 수 있다면 단순한 접근이 훨씬 비용 효율적이다.
- 클라우드 기반 리소스: 클라우드 컴퓨팅 자원을 활용하면 초기 인프라 구축 비용을 줄이고 사용량 기반 과금 모델을 통해 확장성과 유연성을 확보할 수 있다.
- 오픈소스 모델과 도구: 오픈소스 머신러닝 라이브러리와 프레임워크를 활용하면 소프트웨어 개발 비용을 줄이면서도 높은 품질의 에이전트를 구현할 수 있다.

## 아키텍처 디자인 패턴

에이전트 기반 시스템의 아키텍처 설계는 에이전트가 어떤 구조로 이루어져 있으며 환경과 어떻게 상호작용하고 어떤 방식으로 작업을 수행하는지를 결정한다. 아키텍처의 선택은 시스템의 확장성, 유지보수성, 유연성에 직접적인 영향을 미친다.

### 단일 에이전트 아키텍처

가장 단순하고 직관적인 설계 형태. 하나의 에이전트가 시스템의 모든 작업을 관리하고 실행한다. 환경과 직접 소통하며 다른 에이전트의 도움 없이 스스로 의사 결정, 계획 수립, 실행을 수행한다.

구조가 단순하기 때문에 설계, 개발, 배포 용이, 여러 구성 요소 간의 조율, 통신, 동기화 같은 복잡성을 피할 수 있다.

좁은 범위의 업무에 유용하다.

### 멀티 에이전트 아키텍처: 협업, 병렬성, 조율

여러 에이전트가 공동의 목표를 달성하기 위해 협력한다. 작업의 여러 측면을 전문화된 에이전트가 분담하거나 병렬 처리를 통해 효율성과 확장성을 높일 수 있는 상황에 적합하다. 

멀티 에이전트 시스템의 장점

- 협업과 전문화 : 각 에이전트는 특정 작업이나 영역에 특화되도록 설계할 수 있다.
- 병렬성 : 여러 작업을 동시에 수행할 수 있다.
- 확장성 향상 : 시스템이 성장함에 따라 새로운 에이전트를 추가해 더 많은 작업을 처리하거나 부하를 분산할 수 있다.
- 중복성과 복원력 : 여러 에이전트가 독립적으로 작동하므로 에이전트 일부가 작업에 실패하더라도 전체 시스템이 중단되지 않는다.

멀티 에이전트 시스템의 어려움

- 조율과 통신의 복잡성 : 에이전트 간의 통신을 효율적으로 관리해야 한다. 각 에이전트는 중복 작업이나 충돌, 자원 경쟁을 피하기 위해 정보를 교환하고 작동 방식을 조율해야 한다.
- 복잡성 증가 : 멀티 에이전트 시스템은 강력하지만 설계, 개발, 유지보수가 훨씬 어렵다.
- 효울성 저하 : 모든 경우에 해당하지는 않지만 멀티 에이전트 시스템은 토큰 소비가 많아 효율이 낮아질 수 있다.

## 점진적 설계

점진적 설계는 에이전트 개발의 핵심 접근법으로, 초기부터 완벽한 결과물을 목표로 하기보다 작동하는 작은 프로토타입을 만들고 피드백을 반영해 여러 차례에 걸쳐 개선한다.

주요 이점

- 문제의 조기 발견
- 사용자 중심 설계
- 확장성

다음과 같은 상황에서 효과적이다.

- 프로토타입을 빠르게 개발해야 할때
- 테스트하고 피드백을 수집할 때
- 개선하고 반복할때

## 마무리

성공 기준을 명확히 정의하고 모호하거나 과도한 범위는 피하자. 구성 요소만 통합한다고 효과적인 에이전트가 될 수는 없다.
